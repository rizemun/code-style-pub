Работа с баг репортом

Поля баг репорта

Баг должен содержать в себе:
1. Краткое описание
2. Компонент приложения
3. Серьёзность
4. Приоритет
5. Скриншот/видео
  
Другие характеристики опциональны.

В исключительных случаях поле Attachment(скриншот/видео) может быть не заполнено при добавлении бага(баг указывает на проблему, которую не передать через видео или скриншот) //добавить стиль и пример

Если необходимо, то тестировщик должен указать окружение на котором найден баг:

  1. ОС
  2. Браузер
  3. Конкретное разрешение или диапазон

Также могут быть добавлены Шаги для воспроизведения бага, которые помогут разработчику воспроизвести описанный дефект. Шаги представляют собой набор последовательных действий, которые оформлены в виде списка, и выполняются в заданной очередности.

Статусы в баг репорте

Все возможные статусы, которые могут быть использованы при работе с баг репортом:
1. New
2. In progress
3. To test
4. Not fixed
5. Fixed on dev
6. Fixed on prod
7. Fixed
8. Not bug
9. Backlog

Порядок выставление статусов и их предназначение

Каждый новый баг заносится в баг репорт соответствующего проекта со статусом New. Когда разработчик берет в работу баг, то он ставит статус In Progress для соответствующего бага в БР.

Разработчик выставляет статус To test в случае, если баг исправлен и разработчик убедился, что исправления находятся на тестовом сервере, на котором эти исправления будет проверять тестировщик.

После проверки исправленного бага тестировщиком и в случае успешного фикса выставляется статус Fixen on dev или Fixed on prod в зависимости от того есть ли на проекте dev и prod серверы.

Статус Fixed on dev выставляется после статуса To test и означает, что баг исправлен и проверен на dev сервере и ожидает деплоя на prod. После деплоя на прод и в случае успешной починки бага выставляется статус Fixed on prod. Если на проекте есть несколько dev серверов и stage, исправленные баги проверяются при каждом сливании веток на dev, а также на stage.

Тестировщик выставляет статус Not fixed для проверенного бага в случае, если проверенное исправление не решает проблему, указанную в описании бага.

В случае успешной починки бага и его проверке, тестировщик может выставить статус Fixed. Такой статус выставляется, если баг был исправлен при первоначальной разработке, когда основные изменения проходят на dev сервере или если баг исправлен на prod сервере в качестве хотфикса.

Статус Not bug выставляется разработчиком в случае, если найденный баг не является таковым и описание в баге является корректным поведением системы.

Баг может быть добавлен в Backlog. Статус Backlog используется в случае:

  1. Если клиент решил отложить починку бага на неопределенный срок. Починка таких багов может производиться пачками в определенно отведенное время(после апрува клиента)
  2. Если баг был отложен по каким либо причинам командой разработки(например, баги на девайсах, которые пока не релевантны, потому что приложение не ориентировано на мобайл, или какие-то редкие, специфичные баги, на которые нужно затратить большое количество времени на их решение), согласовывается с ПМом.

В случае, если баг многократно(три и больше) попадает в категорию Not fixed, то нужно вынести вопрос ПМу.

Описание бага

В описании бага тестировщик должен кратко и четко изложить проблему, которую он обнаружил.

Описание должны быть:

  1. Понятным
  2. Однозначным
  3. Включать в себя только один недостаток системы
  4. Полным

Пример правильного описания: Приложение зависает, при попытке загрузки видео файла больше 50Мб.

Пример неправильного описания: Плохо работает загрузка видео файла и загрузка картинок.

В случае, если баг по верстке и какой-то элемент на странице не соответствует макету, то к багу нужно прикрепить ссылку на макет, на который ссылается тестировщик. Если макета нет, то можно сослаться на сайт пример или на здравый смысл, при этом в описании бага нужно добавить какие именно стили нужно применить для конкретного элемента на странице.

Если на проекте есть штатный дизайнер, то можно обратиться к нему с вопросом какие стили должны быть для того или иного элемента.

Серьёзность бага

Blocker(блокирующий)
Блокирующая ошибка, приводящая приложение в нерабочее состояние, в результате которого дальнейшая работа с тестируемой системой 
или ее ключевыми функциями становится невозможна. Решение проблемы необходимо для дальнейшего функционирования системы.
Critical(критичный)
Критическая ошибка, неправильно работающая ключевая бизнес логика, дыра в системе безопасности, проблема, приведшая к временному 
падению сервера или приводящая в нерабочее состояние некоторую часть системы, без возможности решения проблемы, 
используя другие входные точки. Решение проблемы необходимо для дальнейшей работы с ключевыми функциями тестируемой системой.
Major(значительный)
Значительная ошибка, часть основной бизнес логики работает некорректно. 
Ошибка не критична или есть возможность для работы с тестируемой функцией, используя другие входные точки.
Minor(не значительный)
Незначительная ошибка, не нарушающая бизнес логику тестируемой части приложения, очевидная проблема пользовательского интерфейса.
Trivial(тривиальный)
Тривиальная ошибка, не касающаяся бизнес логики приложения, плохо воспроизводимая проблема, малозаметная посредствам 
пользовательского интерфейса, проблема сторонних библиотек или сервисов, проблема, не оказывающая никакого влияния 
на общее качество продукта.

Приоритетность бага

High(высокий)
Баг должен быть исправлен как можно быстрее, т.к. его наличие является критическим для проекта
Medium(средний)
Баг должен быть исправлен, его наличие не является критичным, но требует обязательного решения
Low(низкий)
Баг должен быть исправлен, его наличие не является критичным, и не требует срочного решения

Примеры

Каким должен быть Attachment к багу(доделать)

Хороший Attachment имеет такие характеристики и включает в себя:

  1. Показан url страницы на которой найден баг
  2. Указателями(стрелки и т.д.) показано в какой части страницы баг
  3. Добавлен текстовый блок с комментарием по найденному дефекту
  4. Добавленные указатели не перекрывают и не мешают увидеть в чем дефект
  5. На скриншоте/видео показан пример с которым сравнивает тестировщик(макет, в случае с версткой)
  6. Желательно указывать на один дефект за раз.(если указывать сразу на несколько дефектов в пределах одного скриншота можно что-то пропустить при проверке исправлений)

Разобрать на примере плохой и хороший

Типичные действия, которые должен сделать тестировщик и какие вопросы задать, при добавлении его на проект

  1. Есть ли техническая документация по проекту? Где её можно найти?
  2. Есть ли макеты? Макеты для мобильной и таблет версии?
  3. Поддерживаем ли мы мобильную версию приложения?
  4. Уточнить сроки сдачи проекта.
  5. Приоритетность поставленных задач на проекте.
  6. Какие браузеры мы поддерживаем на проекте?
  7. Уточнить за доступы(будь то доступы к админ панеле, базе данных и т.д.), которые нужны для тестирования.
  8. Какую баг трекинговую систему используем?
  9. На каком языке оформляется баг репорт?
  10. Есть ли API документация по проекту?(если нет, но тестировать API нужно, то нужно обращаться к разработчику лично)
  11. Попросить добавить его к проекту в Toggl и уточнить является ли потраченное время на проекте billable($)

Затем спланировать работу на проекте исходя из полученных ответов. Подготовить тестовые данные, чеклисты, составить тест кейсы( если это нужно на проекте и отведенное время на тестирование позволяет спокойно выполнить эти задачи).

Инструменты тестировщика(доделать)

Инструменты/приложения/расширения, которые помогут тестировщику при тестировании верстки:

  1. Chrome DevTools.
  2. Tape(расширение для Google Chrome), аналог линейки, помогает мерить расстояния между блоками.
  3. PerfectPixel(расширение для Google Chrome), позволяет накладывать макет на странице в браузере.

Особенное внимание стоит уделять(!):

  1. Тестированию верстки при ресайзе страницы
  2. Тестированию верстки в разных браузерах(Chrome, Firefox, Safari)
  3. Тестированию верстки на разных ОС(Windows, Mac)
  4. Тестирование на мобильных устройствах(Android, iOS)

Инструменты/приложения/расширения, которые помогут тестировщику при составлении баг репорта

  1. Bird Eats Bug: Technical Screen Recording (расширение для Google Chrome), позволяет записывать действия на странице.
  2. Lightshot(приложение), позволяет делать скриншоты экрана и добавлять сопровождающую информацию на скриншот.

Примечание: скриншоты и видео в баг репорт лучше добавлять файлом, а не ссылкой!!!

  1. text

Инструменты/приложения/расширения, которые помогут тестировщику при тестировании API

  1. Postman(приложение), приложение для работы с запросами, позволяет отправлять запросы с тестовыми данными, получать ответы(и многое другое)
  2. Proxyman(приложение), сниффер траффика, позволяет перехватывать запросы и подменять ответы(и многое другое)
