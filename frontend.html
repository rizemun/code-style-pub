CodeStyle

Для повышения качества кода нужно соблюдать некоторые правила при разработке, одно из которых, очень важное, это единообразие. Единообразие в именовании, форматировании кода и подходах к решению задач это то, что позволяет на дистанции ускорить разработку, ввиду уменьшения порога входа новых разработчиков в существующие проекты или развертывание новых. Полная унификация невозможна ввиду различных применяемых технологий и стеков этих технологий, а также проектов с ними. Можно лишь сделать это в рамках отдельных технологий. Далее будут описаны принципы единообразия, которых крайне здорово было бы придерживаться при работе в рамках рабочих компетенций с языками программирования и другими сущностями вовлеченными в процесс разработки.

Общее

Файловая структура компонента

  1. Реэкспорт в index файлы в корне компонентов - более не использовать в новых проектах, по возможности стараться переписывать. В большом количестве такие файлы плохо влияют на тришейкинг и сбоки webpack.
  2. Утилитарные функции/хелперы выносятся в отдельную директорию helpers/{helperName}.js.
  3. Кастомные хуки react также выносим в отдельную директорию hooks/{hookName}.js.
  4. Для сборных сущностей, таких как файлы в директориях helpers, hooks а так же svg компонентов, рекомендуется использовать реэкспорт в index для улучшения читабельности кода.
  5. Компоненты создаются всегда в своей именной папке с большой буквы ../Button/index.jsx. Остальные функции с маленькой буквы. Допускается разбиение компонентов по смысловым группам, например - разметка страницы, специфичные для страниц компоненты,общие компоненты.
  6. В React в директории с компонентом рекомендуется хранить файл со стилями с названием, соответствующем компоненту, подробнее см в части про стили.
  7. TS: Объявление типов и интерфейсов выносятся в отдельный файл interfaces.ts. Для понимания в какой именной файл выносить типизацию, необходимо руководствоваться первоисточником информации для определения этой типизации. Например хелперы, как правило, не являются источником данных, тк они сами нужны для чего-то, для них типы выносятся на уровень выше.
  8. TS: Объявления констант и enum выносятся в отдельный файл constants.ts

Именование переменных

  • Для именования js переменных используем lowerCamelCase
  • Для именования css классов используем BEM. В качестве разделителей между блоком и элементом используем __, между элементом и модификатором _.
  • Любое объявление переменных необходимо называть полностью без сокращений. Исключения - однозначно понятные большинству сокращения ev от event и i от index.
function generateUniqKey(val: string) { ... }       // боль
function generateUniqKey(value: string) { ... }     // ок
 
const result = entities.map((e, i) => ...)          // боль
const r = entities.map((ent, i) => ...)             // боль
const result = entities.map((entity, index) => ...) // ок
const result = entities.map((entity, i) => ...)     // ок
  • Называя массивы с окончанием list не использовать значение перед во множественном числе
const usersList = [ ... ]   // боль
const userList = [ ... ]    // ок
  • Переменные необходимо называть начиная с существительных - variableForExample
  • Styled-components необходимо называть заканчивая "Styled" - ButtonStyled
  • Константы для примитивных типов данных необходимо называть через upperCase
const TITLE_LENGTH_MAX = 100 
const DEFAULT_ACTION = "start"
  • Булевые переменные необходимо называть начиная с "is" или "has" - isShow/hasProperty. Исключения это нейминг устоявшихся аттрибутов типа disabled/required/..
  • Функции необходимо называть начиная с глаголов - loadSomeData
  • Обработчики событий необходимо называть с "handle" - handleInputChange
  • Пропсы для обработчиков необходимо называть с "on" - onChange={handleChangeInput}
  • Утилиты/хелперы необходимо называть с семантичных глаголов "get/create/generate" и тд - getFieldValue/generateUniqKeys
  • Interface необходимо называть начиная с "I" и большой буквы
interface IButtonProps { ... }
  • Типы необходимо называть с "T" большой буквы
type TButtonSizes = "s" | "m" ...
  • Enum необходимо называть с "E" большой буквы + перечисления enum c большой буквы
enum EContentRoutes { 
  MainRout = "/main" 
}
  • Если необходимо пропустить какой-либо ненужный параметр в функции, то следует использовать в качестве имени переменной _.

Лучшие практики:

  • Лучше избегать смешивания абстрактного нейминга и бизнеса
const userItems;                // боль
const userArray;                // боль
const userPermissionsObject;    // тоже боль
 
const user;                     // ок
const userList;                 // ок

Общее соглашение по коду

  • В рендере (верстке) использовать стрелочные/анонимные функции запрещается, лучше создать дополнительный обработчик, чтобы не создавать лишнюю нагрузку на память
// боль
const onChangeValue = (event, id) => { ... }
 
<Button onClick={(event) => onChangeValue(event, id)} />
 
// так лучше
const onChangeValue = useCallback((id) => (event) => { ... }, [id])
  
<Button onClick={onChangeValue(id)} />
  • Желательно деструктурировать большое кол-во элементов из объекта/массива в несколько подходов по категориям (обработчики/булевые/переменные), чтобы избежать "PropsWall"
// боль
const {
  prop1,
  prop2,
  prop3,
  prop4,
} = props 
 
// ок
const { prop1, prop2, prop3 } = props
const { prop4, prop5 } = props
  • Инлайного деструктурировать props в компоненте можно только, когда они умещаются в одну строку, иначе необходимо деструктурировать, как описано выше
// боль
export const ExmapleComponent: FC<IExampleComponent> = observe(({
prop1,
prop2,
prop3,
prop4,
}) => { ... })
 
// ок
export const ExmapleComponent: FC<IExampleComponent> = observe((props) => {
  const { prop1, prop2, prop3, prop4} = props
    ...
})
  • Мультитернарные выражения запрещены
// боль
isYes
  ? isYes2
    ? do1
    : do2
  : do3
  • Использовать блок if else, если условий более двух, для одного блока использовать только if
// боль
if (isYes) {
  do1...
} else {
  do2...
}
 
// ок
if (isYes) {
  do1...
  return
}
 
do2...
  • Блоки if пишем с фигурными скобками
// боль
if (isYes) do...
 
// ок
if (isYes) {
  do...
}
  • Именовать компоненты и компоненты styled-component необходимо максимально семантично, не стоит плодить ButtonStyled по проекту, всегда можно уточнить - AddPersonButtonStyled
  • Для переиспользования styled-component у него всегда должен быть объявлен пропс className - иначе не будет работать
  • Если сроки поджимают и задача === крит, а решение получилось костыльное - необходимо создать задачу техдолгом, оставить TODO c ее номером/id и автором костыля :)
/** TODO (номер таски из Jira/Trello) Вася Костыль - вынужденная мера, нужно лучше */
  • Все утилиты/хелперы должны быть чистыми функциями
  • Каждая утилита/хелпер должна жить в своем именном файлке
// боль
Product.helpers.ts
export function getSome1() { ... }
export function getSome2() { ... }
export function getSome3() { ... }
 
// ок
helpers/getSome1.ts
export function getSome1() { ... }
helpers/getSome2.ts
export function getSome2() { ... }
helpers/getSome3.ts
export function getSome3() { ... }
  • Каждая утилита/хелпера должна быть объявлена через function, чтобы иметь возможность прокинуть в нее дженерики (стрелочная функция не может принимать дженерик без явной типизации)
export const getSome = () => ... // боль 
export function getSome() { ... } // ок
  • Экспорт утилит/хелперов должен быть именной
export default function() { ... } // боль
export function getSome() { ... } // ок

React

  1. useMemo должен использоваться для действительно тяжелых вычислений или передачи объектов/массивов через пропсы дочерним компонентам
  2. useCallback должен использоваться для функций передаваемых дочерним компонентам
  3. Оборачивать компонент в memo необходимо только view компоненты c примитивными типами данных в пропсах. Если во view компонент еще передаются объекты/массивы, то с компонентом что-то не так, но в этом случае такие данные должны быть промемоизированны родителем`
  4. Создавать переменные рендера в компоненте запрещено - лучше создать отдельный компонент, чтобы не усложнять оптимизацию рендера
// боль
const FunctionComponent = () => {
  const Spinner = () => <Spinner {...props} />
  
  return (
    {Spinner()}
  )
}
  1. Любые вычисления, объекты/массивы конфигураций, константы, типы, интерфейсы и тд - все выносится из компонент
  2. Экспорт компонент должен быть именной

TypeScript

  1. Во всех функциях, которые что-то возвращают должно быть протипизированно возвращаемое значение, если значение явно протипизированно
  2. Необходимо использовать интерфейсы и типы осмысленно, не стоит плодить все одними интерфейсами
  • Типом является чаще всего: 1 параметр из 3 и более типов; функция; перечисления;
type TItemValue = string | number | null
type TDelivery = 'courier' | 'postmate'
  1. Необходимо использовать утилитарные типы
type TStringsObject = { [key: string]: string } // боль
type TStringsObject = Record<string, string> // ок
  1. В интерфейсах допустим только 1 уровень вложенности
// боль
interface ISomeProps {
  item1: {
    prop1: string
  }
}
 
// ок
interface IItem {
  prop1: string
}
 
interface ISomeProps {
  item1: IItem
}
  1. Очевидно не стоит явно указывать тип undefined, можно сделать параметр необязательным
// боль
interface ISomeProps {
  prop1: string | undefined
}
 
// ок
interface ISomeProps {
  prop1?: string
}
  1. Все утилиты/хелперы должны быть протипизированы, any/unknown/IObjectAny/кастинги через "as" и прочее строго запрещены! Если неизвестно какой тип данных приходит в чистую функцию или наоборот, ожидается любой - используйте дженерик
// боль
export function getId(data: any): string {
  return `${data?.id ?? ''}`;
}
 
// ок
export interface IData {
  id?: string | number;
}
 
export function getId<D extends IData>(data: D): string {
  return `${data?.id ?? ''}`;
}
  1. Any types допустимы только для данных из апи/библиотек или любые другие, о происхождении которых невозможно знать. За пределы файла такие данные с эни типами не должны выходить

Действия при создании проекта

Создать файл README.md или отредактировать сгенерированный, добавив информацию о проекте, такую как: • название проекта • краткое описание, если из названия проекта не ясно его предназначение

• вся необходимая информация для развертывания (установки) проекта и запуска/тестирования, обязательно самим протестировать с нуля разворачивание в новой папке. • Если в проекте используются нестандартные библиотеки или архитектурные подходы, то их тоже нужно указывать. Причем желательно указывать ссылки на их репозитории, чтобы человеку поставленному на этот проект, но не работавшему с используемыми технологиями, не нужно было искать их в гугле. • Если проект из репозитория уже где-то развернут (например, на github pages), то следует указать на него ссылку. В проекте должно быть использовано по минимуму глобальных библиотек, все они должны быть перечислены в Readme (webpack, karma, sanity и все, что может быть поставлено через npm, точно не должно быть глобальным)

Общие настройки

Git

В git стоит настроить пользователя, для корректного отображения. Рекомендуется использовать формат nSurname /NSurname/nsurname, где n - первая буква имени, а Surname, соответственно, фамилия. Допустимы и другие варианты. Так же стоит указать почту.

Проверка текущего пользователя:

git config user.name
git config user.email

Изменение данных:

git config --global user.name "John Doe"
git config --global user.email johndoe@example.com

На windows компьютерах также возникает проблема с символами переноса строки, многие файлы попадают в коммит из-за того, что git по умолчанию приводит символ переноса строки к системному. Чтобы настроить приведение символов переноса строки нужно воспользоваться настройкой File ⇒ Settings ⇒ Editor ⇒ Code style ⇒ General ⇒ Line separator в значение Unix and macOS(\n) и изменением конфига git:

git config --global core.autocrlf false

.editorconfig

Файл расположенный в корне проекта используется для единообразной общей настройки разных IDE и редакторов.

Действия разработчика при включении в проект (фронт)

  • Спросить на входе, под какие браузеры верстаем (обязательно).
  • Спросить, есть ли адаптив, если нет — то под какие разрешения верстать и что делать для других экранов.
  • при работе использовать локальный сервер а не открытие файла в браузере (Запускать сайт через http://localhost, а не через file:///c:\test.html). Причина - иначе могут быть проблемы с элементами, вставляемыми через iframe (например, лайки от фб, вк и т. д.), и при работе скриптов, которые отправляют HTTP-запросы, плюс еще могут быть вообще неожиданные и неизвестные нам проблемы.
  • Заранее четко узнать, что является фиксированным результатом проекта — развернутое на нашем сервере production-окружение или только git-репозиторий, а то и zip-архив, отправленный по почте.
  • Всегда помните: если вы можете решить проблемы каким-то способом, это не значит, что способ действительно стоит применять. И в верстке, и в программировании часто возникают ситуации, когда можно написать вполне рабочий код, но сделать это ненадежным способом, либо используя откровенные хаки, неочевидные другим разработчикам, либо просто оформив все некрасивым способом. Если перед вами появилась проблема, то нужно не только решить ее, но и сделать решение читабельным, масштабируемым и надежным. Во всех случаях, когда появляются сомнения, лучше спросить более опытных товарищей - это и вам позволит быстрее вырасти, и проекту получить более качественный код.
