Первые шаги на проекте

Типичные действия, которые должен сделать тестировщик и какие вопросы задать, при добавлении его на проект

  1. Уточнить сроки сдачи проекта.
  2. Приоритетность поставленных задач на проекте.
  3. Есть ли техническая документация по проекту? Где её можно найти?
  4. Есть ли макеты? Макеты для мобильной и таблет версии?
  5. Поддерживаем ли мы мобильную версию приложения?
  6. Какие браузеры мы поддерживаем на проекте?
  7. Уточнить за доступы(будь то доступы к админ панеле, базе данных и т.д.), которые нужны для тестирования.
  8. Какую баг трекинговую систему используем?
  9. На каком языке оформляется баг репорт и другая тестовая документация?
  10. Есть ли API документация по проекту?(если нет, но тестировать API нужно, то нужно обращаться к разработчику лично)
  11. Попросить добавить себя к проекту в Toggl и уточнить является ли потраченное время на проекте billable($)

Затем спланировать работу на проекте исходя из полученных ответов. Подготовить тестовые данные, чеклисты, составить тест кейсы(если это нужно на проекте и отведенное время на тестирование позволяет спокойно выполнить эти задачи).

Виды тестирования применяемые на проекты

Smoke тестирование
Это короткий цикл тестов, который направлен на проверку только основных функций определенных бизнес логикой приложения.
Такой вид тестирования обычно применяется после обновления production сервера и направлен на проверку только основных функций
приложения. Такое тестирование должно занимать около 15 минут.

Какие проверки будут входить в Smoke тест определяет тестировщик. Итоговый набор тестов согласовывается с ПМом на проекте.
Часто в такой набор тестов будет входить: регистрация, авторизация, оплата, создание сущностей в приложении и другие основные функции
приложений, в зависимости от его назначения.

Smoke тестирование проводится раньше, чем проверка добавленных задач на production сервер, это нужно для того, чтобы убедиться,
что основные функции приложения работают стабильно.
Регресионное тестирование
Это набор тестов, направленный на проверку всех компонентов приложения после крупных обновлений. Производится
такое тестирование после значительных изменений в приложении(например, добавлении новых компонент или крупную переработку старых) и проходит
на dev/stage сервере. Основная цель такого вида тестирования - это проверить не сломалось ли что-то в приложении после обновления, при этом
неважно вносились ли изменения в какой-то конкретный компонент, проверить нужно всё.
Ретест
Набор тестов, который направлен на проверку старый багов на предмет фикса. Проводится в случае, если фича/функциональность
уже имела дефекты, и эти дефекты были недавно исправлены.

(!)Ретест охватывает только тот функционал, который был исправлен

Инструменты тестировщика(дополнить)

Инструменты/приложения/расширения, которые помогут тестировщику при тестировании верстки, полей ввода, текстовых полей:

  1. Chrome DevTools.
  2. Tape (расширение для Google Chrome), аналог линейки, помогает измерять расстояния между блоками.
  3. PerfectPixel (расширение для Google Chrome), позволяет накладывать макет на странице в браузере.
  4. Grid Ruler(расширение для Google Chrome), аналог линейки, помогает измерять расстояния между блоками.
  5. Bug Magnet (расширение для Google Chrome), помощник по исследовательскому тестированию для Chrome, работает с полями ввода, текстовыми областями, редактируемыми элементами DIV. Помогает с проверками граничных значений

При тестировании верстки особенное внимание стоит уделять(!):

  1. Тестированию верстки при ресайзе страницы
  2. Тестированию верстки в разных браузерах(Chrome, Firefox, Safari)
  3. Тестированию верстки на разных ОС(Windows, MacOS)
  4. Тестирование на мобильных устройствах(Android, iOS)

Инструменты/приложения/расширения, которые помогут тестировщику при составлении баг репорта

  1. Bird Eats Bug: Technical Screen Recording (расширение для Google Chrome), позволяет записывать видео действий на странице.
  2. Cloudapp(платно), позволяет делать скриншоты экрана, записывать видео и другое.
  3. Lightshot(приложение), позволяет делать скриншоты экрана и добавлять сопровождающую информацию на скриншот.

Примечание: скриншоты и видео в баг репорт лучше добавлять файлом, а не ссылкой!!!

Баг трекинговые системы с которыми придется работать:

  1. Trello
  2. JIRA

Инструменты/приложения/расширения, которые помогут тестировщику при тестировании API

  1. Postman(приложение), приложение для работы с запросами, позволяет отправлять запросы с тестовыми данными, получать ответы(и многое другое).
  2. Proxyman(приложение, MacOS), сниффер траффика, позволяет перехватывать запросы и модифицировать их(и многое другое).
  3. Proxyman(приложение, Windows(beta)).
  4. Swagger UI Пользовательский интерфейс Swagger — это набор ресурсов HTML, JavaScript и CSS, которые динамически генерируют красивую документацию из совместимого с Swagger API.
  5. Charles(приложение), сниффер траффика, позволяет перехватывать запросы и модифицировать их(и многое другое).

Инструменты/приложения/расширения, которые помогут тестировщику при тестировании мобильных приложений

  1. Xcode
  2. Android Studio
  3. Charles

Подготовка и использование тестовых данных при тестировании

Часто в приложениях приходится иметь дело с функционалом, где в том или ином виде, пользователю нужно загружать данные(картинки, видео, файлы), которые должны точно соответствовать требованиям указанными разработчиком. При тестировании таких функциональностей тестировщик должен четко понимать на каком наборе тестовых данных нужно проводить тестирование, чтобы охватить как можно больше возможных пользовательских кейсов. К тестовым данным есть несколько требований:

  1. Уникальность(контента) в пределах тестового набора
  2. Разнообразие данных по используемым доступным расширениям файлов
  3. Разнообразие данных по используемым доступным разрешениям файлов
  4. Разнообразие данных по размеру файлов
  5. Контент должен быть нейтральным. Животные, природа и т.д. - для картинок и видео, для файлов - ничего конфиденциального. (особенно важно придерживаться этого пункта, если тестирование проходит на сервере к которому клиент имеет непосредственный доступ)

(!) Ни в коем случае не использовать личные фото/видео/файлы или скриншоты/видео с других проектов при тестировании
